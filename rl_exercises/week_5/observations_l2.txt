1. We tried max_episode_steps = 50, 100, 200, and 500 on CartPole-v1. With shorter trajectories, the agent was able to reach the maximum reward more reliably. When max_episode_steps was set to 50, the evaluation reward reached 50.0 with almost no variance. For 100 and 200 steps, the agent could also reach the maximum return, but the training and evaluation curves became not very stable. With the full 500-step training, the agent sometimes reached 500 very early on, but subsequently crashed on several times. This indicates that longer trajectories increase the difficulty of this task. Long trajectories brings higher rewards, but on the other hand, they increase the variance in hte policy gradient estimates. Vice versa, shorter trajectories are more easier and more stable, but artificially limit the maximum performance that can be achieved.

2. As LunarLander-v2 is marked as deprecated, we decided to try using LunarLander-v3. On LunarLander-v3, using the same architecture and learning rate used in CartPole-v1 did not yield same stable learning results. The training process was extremely unstable and the evaluation return fluctuated heavily between positive and strongly negative values. At some points, the agent achieved evaluation returns exceeding 200, indicating that the policy had temporarily succeeded. However, the performance later crashed out again, sometimes even falling below -300.This indicates that the learning rate may have been too high, and that the vanilla algorithm suffers from high variance on more difficult environments. It can be inferred that LunarLander-v3 has a more complex state space and reward structure than CartPole-v1. Thus, the previous network architecture and learning rate work cannot work well for LunarLander-v3.
